## 0. v3에서 무엇이 바뀌었나

v2는 한 단체용 기록 도구를 SaaS 플랫폼 수준으로 설계해 복잡했다. v3는 핵심 가치(여러 사람이 일정 단위로 기록을 쌓고, 그것을 짧은 스토리 영상의 대본으로 정리한다)에 집중하고, 나머지는 MVP 이후로 미룬다.

주요 변경:

```text
역할 4개 -> 2개 (관리자 / 업로더), 보기는 공유 링크로만
가입 승인 파이프라인 제거 -> 초대 링크 클릭 시 자동 합류
기록 단위를 "일정(장면)"으로 명확화
업로더별 "기여 묶음" 개념 도입 (본인 것만 수정)
AI 기능 7종 -> 1종 (스토리 합치기 + 길이 요약)만 MVP
영상 길이를 "계산되는 값"으로 정의
BGM / PDF / 이미지 생성 / 다단계 백업 / CI 자동배포 -> MVP 이후
```

## 1. 제품 개요

MemoryFlow는 여행, 수련회, 행사, 선교캠프처럼 여러 사람이 함께 참여하는 프로젝트에서 사진, 영상, 스토리를 일정 단위로 수집하고, 이를 짧은 스토리 영상의 대본으로 정리하는 웹 애플리케이션이다.

프로젝트를 잘 아는 소수의 업로더가 각 일정에 사진과 자기 관점의 스토리를 남긴다. 관리자는 모인 기록을 일정 단위로 정리하고, 여러 관점의 스토리를 하나의 내레이션으로 합쳐 스토리북을 완성한 뒤 승인한다. 승인된 스토리북은 공유 링크로 외부에 공개하거나, 영상 제작용 패키지로 내보낸다.

영상 렌더링은 MemoryFlow에서 직접 하지 않는다. 승인된 스토리북을 제작용 패키지로 내보내 외부에서 영상을 만들고, 완성된 MP4를 관리자가 다시 업로드한다. 제작 방법은 별도 VideoFlow 문서에서 다룬다. (11장 참고)

MemoryFlow 웹앱의 기술 스택은 별도 재검토 후 확정한다. 서버에서 로컬 파일 스토리지 기반으로 운영하는 구조를 전제로 한다.

## 2. 핵심 흐름

```text
관리자가 프로젝트 생성 (날짜 -> Day 자동 생성)
-> 관리자가 Day별 세부일정 설계 (대본의 목차)
-> 관리자가 프로젝트별 초대 링크 발급
-> 업로더가 링크 클릭 -> 자동 합류
-> 업로더들이 각 일정에 사진/영상 + 자기 스토리 업로드
-> 관리자가 스토리북 편집 (일정별 사진 선별 + 스토리 합치기)
-> AI가 일정별 스토리를 한 단락 내레이션으로 합침
-> 관리자가 내레이션을 영상 길이에 맞게 다듬고 확정
-> 관리자가 스토리북 승인 (전체 잠금)
-> 공유 링크 발급 (외부 열람)
-> (선택) 영상 제작용 패키지 내보내기 -> 외부에서 영상 제작 -> 완성 MP4 업로드
```

## 3. 역할과 권한

역할은 둘이다. "보기"는 역할이 아니라 공유 링크로 처리한다.

### 관리자

운영과 산출물 전체를 담당한다. (v2의 슈퍼 관리자 + 프로젝트 관리자를 통합)

```text
프로젝트 생성/삭제
세부일정 설계와 관리
프로젝트별 초대 링크 발급/무효화
멤버 목록 관리 (사후 내보내기/비활성화)
스토리북 편집: 사진 선별, 순서 조정, 스토리 합치기/수정
AI 스토리 합치기 실행
스토리북 승인 / 승인 잠금 해제
공유 링크 생성/관리
최종 영상 업로드/삭제
영상 제작용 패키지 내보내기
```

> 시스템 전체를 관장하는 권한(다른 관리자 임명 등)은 별도 역할로 만들지 않고, 특정 계정에 부여하는 플래그로 처리한다. MVP에서는 관리자 = 운영자 1인 또는 소수로 충분하다.

### 업로더

배정된 프로젝트의 일정에 사진/영상과 스토리를 올린다.

```text
배정된 프로젝트의 일정에 사진/영상 + 스토리 업로드
본인이 올린 기여(사진/영상/스토리)만 수정/삭제 (승인 전까지)
승인된 스토리북과 최종 영상 열람
```

스토리북 승인 후 업로더의 기여는 모두 읽기 전용으로 잠긴다.

### 보기 (공유 링크 / 역할 아님)

로그인 없이 `/share/:token`으로 접근한다.

```text
승인된 스토리북 열람
공개된 최종 영상 시청
허용된 공유 파일 다운로드
```

업로더도 결과물을 볼 때는 이 공유 링크로 본다. 앱 안에 별도 열람 전용 화면을 두지 않는다.

허용하지 않음: 로그인, 업로드, 댓글, 좋아요, 편집, 비공개 데이터 조회.

## 4. 인증과 가입

별도의 가입 승인(pending) 단계가 없다. 관리자가 발급한 초대 링크 자체가 신뢰의 근거다.

```text
관리자가 프로젝트에서 초대 링크 생성 (/join/:token)
-> 단톡방/문자로 공유
-> 받은 사람이 링크 클릭
-> (계정 없으면) 이름 + 전화번호 + 비밀번호 입력해 가입
-> 자동으로 해당 프로젝트 업로더로 합류
-> 바로 업로드 시작
```

초대 링크 단위는 **프로젝트별**이다. 한 사람이 여러 프로젝트에 참여하면 각 프로젝트의 링크를 받아 합류한다. 계정은 하나로 유지된다.

### 로그인 식별자

이름은 유일하지 않으므로(동명이인) 로그인 식별자가 될 수 없다. **로그인 식별자는 전화번호다.**

```text
가입 입력: 이름 + 전화번호 + 비밀번호
로그인 입력: 전화번호 + 비밀번호
전화번호는 시스템 전체에서 유일 (중복 가입 차단)
이름은 표시용 (멤버 목록·기여 표기에 사용, 중복 허용)
```

MVP에서는 전화번호 인증(SMS)을 하지 않는다. 초대 링크가 신뢰의 근거이므로, 전화번호는 "고유한 로그인 ID" 역할만 한다. SMS 인증·이메일은 Phase 2.

링크 안전장치:

```text
만료 시간 (기본값 설정, 만료 후 무효)
관리자가 언제든 무효화 가능
들어온 사람은 멤버 목록에 자동 표시
관리자가 사후에 누구든 내보내기/비활성화 가능
```

사용자 상태는 둘이다.

```text
active
inactive
```

비밀번호 재설정은 MVP에서 관리자가 임시 비밀번호를 설정하는 방식으로 처리한다. 이메일 기반 재설정은 Phase 2.

## 5. 프로젝트와 일정 구조

프로젝트 생성 항목:

```text
프로젝트명
소속/단체명
설명
대표 이미지 (관리자가 직접 업로드. AI 생성은 Phase 2)
시작일
종료일
상태
```

프로젝트가 생성되면 시작일~종료일 기준으로 Day가 자동 생성된다.

```text
2026-07-15 ~ 2026-07-17
-> Day 1 / 2026-07-15
-> Day 2 / 2026-07-16
-> Day 3 / 2026-07-17
```

각 Day에는 세부일정이 있다. **세부일정은 단순 메타데이터가 아니라 영상 대본의 한 장면(scene) 단위다.** 그래서 관리자가 미리 잘 설계할수록 좋다.

세부일정 항목:

```text
시간
제목
장소
분류
정렬 순서
사진 노출시간(초)   <- 영상용. 미설정 시 프로젝트 기본값 사용 (8장 참고)
```

모든 업로드는 프로젝트 / Day / 세부일정에 연결되어야 한다. 업로더는 일정 목록(관리자가 미리 만든 것)에서 고르기만 하면 되므로 부담이 적다.

## 6. 기록 모델 (핵심)

### 멘탈 모델: 일정 = 장면, 그 안에 업로더별 기여가 쌓인다

기록의 단위는 "사진 한 장"이 아니라 "일정 하나"다. 사진/영상은 그 일정의 의미를 보여주는 비주얼 소재이고, 스토리가 그 일정의 의미를 담는다.

같은 일정이라도 각 업로더는 자기 사진/영상 + 자기 스토리를 **하나의 기여 묶음(contribution)**으로 올린다. 그래야 본인이 쓴 글을 본인이 수정할 수 있다(소유 명확).

```text
Day 2 - 저녁 집회  (장면)
├─ [A의 기여]  사진 3장 + "은혜로운 찬양 시간이었다"      <- A만 수정
├─ [B의 기여]  사진 2장 + "OO형제가 눈물을 흘렸다"        <- B만 수정
└─ [C의 기여]  영상 1개 + "말씀 후 통성기도가 뜨거웠다"   <- C만 수정
```

### 사진 중복

업로더가 자기 기여를 올릴 때, 그 일정에 이미 올라온 다른 사람의 사진/영상이 보인다. 같은 장면이면 안 올리면 된다(보여주기만 하고 차단하지 않음). 더 좋은 컷이 있으면 추가 가능하다. 어차피 영상은 일정 단위로 사진이 흐르므로, 약간의 여분은 관리자가 승인 단계에서 선별한다.

> 정확히 동일한 파일만 해시로 가볍게 차단하는 것은 선택적으로 가능(구현 부담 적음). 이미지 유사도 기반 자동 중복 감지는 하지 않는다.

### 글-사진 연결

스토리와 사진을 개별적으로 연결하지 않는다. 영상이 일정 단위로 흐르기 때문에, "같은 일정에 속한다"는 것만으로 스토리와 사진이 함께 흐른다. 별도의 연결 테이블이 필요 없다.

## 7. 스토리북

스토리북은 프로젝트의 최종 기록물이자 영상 대본이다. Day -> 세부일정(장면) -> 그 안의 사진/영상 + 합쳐진 내레이션 순으로 구성된다.

### 편집 (관리자)

장면 하나를 열면 모인 기여가 업로더 구분이 보이는 채로 펼쳐진다. 관리자가 하는 일은 셋이다.

```text
1. 사진/영상 선별   - 포함/제외 토글 (제외해도 흐리게 남아 되돌리기 가능)
2. 순서 조정        - 드래그로 장면 내 순서 변경
3. 스토리 합치기    - 여러 관점 스토리를 한 단락 내레이션으로 (AI 보조 + 사람 최종 수정)
```

### 편집 화면 개념

```text
[Day 2 - 저녁 집회]   예상 길이: 23초

사진/영상 (선별)
[●][●][○][●][●]  + 🎬(8초)
사진 노출: [3]초   <- 이 장면 덮어쓰기 (미설정 시 프로젝트 기본)

스토리 (관점 A·B·C)
[A] 은혜로운 찬양 시간...
[B] OO형제가 눈물을...
[C] 말씀 후 통성기도가...
        [대본 초안 생성]   <- AI 호출

내레이션 (합쳐진 결과, 편집 가능)
┌──────────────────────────────┐
│ 은혜로운 찬양이 흐르는 가운데    │  23초 분량 ≈ 약 120자
│ OO형제가 눈물을 흘렸고...       │  (현재 145자 - 조금 길어요)
└──────────────────────────────┘
        [길이에 맞게 요약]   <- AI 보조
```

### 편집 잠금 (승인 전, 선택)

관리자가 스토리를 합치고 다듬는 동안에도 업로더가 새 기여를 계속 올릴 수 있다. 관리자가 본 입력과 승인 시점의 입력이 달라지는 것을 막기 위해, 승인과 별개로 **"편집 잠금"**을 둔다.

```text
관리자가 편집 잠금 ON
-> 업로더의 신규 업로드/수정 차단 (관리자는 계속 편집)
-> 잠금 이후 들어온 변경이 없으므로 관리자가 본 상태 = 승인할 상태
-> 합치기/다듬기 완료 후 승인
```

편집 잠금은 선택이다. 소규모로 충분히 합의됐다면 바로 승인해도 된다. 승인은 편집 잠금을 포함한다(승인 시 자동으로 모든 기여가 잠긴다).

### 승인

승인하면 해당 프로젝트의 모든 기여(업로드/수정)가 잠긴다. 승인된 스토리북은 관리자만 잠금 해제할 수 있다.

정적 HTML 내보내기는 MVP 제외. 스토리북 웹페이지가 HTML 열람 역할을 대신한다.

## 8. 영상 길이 모델

영상 길이는 관리자가 직접 입력하는 값이 아니라, 내용물에서 **계산되는 값**이다.

```text
한 장면의 길이
= (포함된 사진 수 × 사진 노출시간)
+ 포함된 영상 클립들의 길이 (클립은 자체 길이 사용)

전체 영상 길이 = 모든 장면 길이의 합
```

예시:

```text
Day 2 - 저녁 집회
- 사진 5장 × 3초 = 15초
- 영상 클립 1개 = 8초
-> 장면 길이 23초
... 모든 장면 합산 -> 전체 약 2분 40초
```

사진 노출시간 설정:

```text
프로젝트 기본 사진 노출시간 (예: 3초)
└─ 장면별로 덮어쓰기 가능 (스토리북 편집 시)
```

### 내레이션과 길이 맞추기

길이 기준은 **화면(장면 길이)**이다. 장면 길이가 정해지면, 그에 맞는 내레이션 분량이 산출된다(읽기 속도 기준, 대략 분당 300~350자 -> 초당 글자 수로 환산). 내레이션이 길면 그 길이에 맞게 AI가 요약하고, 사람이 최종 다듬는다.

```text
장면 길이(초) -> 적정 내레이션 글자 수 가이드 표시
내레이션이 길면 -> "길이에 맞게 요약" (AI 보조) -> 사람이 확정
```

길이 가이드는 **강제가 아니라 제안**이다. 글자 수가 초과해도 저장·승인을 막지 않는다. 좋은 스토리가 사진 수에 의해 강제로 잘려나가면 안 되기 때문이다.

내레이션이 장면 길이보다 길 때 관리자에게는 두 개의 레버가 있다. 어느 쪽도 강요하지 않는다.

```text
레버 1 (화면을 늘림)  - 사진을 더 포함하거나 노출시간을 늘려 장면을 길게
레버 2 (글을 줄임)    - "길이에 맞게 요약"으로 내레이션을 짧게
```

즉 길이는 "화면 우선"의 단일 규칙이 아니라, 관리자가 화면과 글 사이에서 조율하는 값이다. 가이드는 그 조율을 돕는 신호일 뿐이다.

## 9. AI 기능 (MVP)

MVP의 AI는 사실상 하나다. 서버의 AI 백엔드를 통해 실행한다. (구체적인 모델·실행 도구는 기술 스택 재검토에서 확정)

```text
✅ 스토리 합치기 + 길이 요약
   - 한 일정에 모인 여러 관점의 스토리를 한 단락 내레이션으로 합침
   - 자막/내레이션 초안 작성을 겸함 (v2의 별도 항목 흡수)
   - 장면 길이에 맞게 분량 조절(요약)
```

### 실행 방식 (가볍게)

v2의 무거운 작업 큐(ai_jobs) + 백그라운드 Worker 구조는 쓰지 않는다. 관리자가 편집 화면에서 버튼을 누르면 그 자리에서 동기 호출하고, 몇 초 뒤 결과가 화면에 뜬다.

```text
관리자가 "대본 초안 생성" / "길이에 맞게 요약" 클릭
-> 서버가 AI 백엔드 동기 호출
-> 결과(초안)를 화면에 표시
-> 관리자가 다듬어 확정
```

동기 호출이지만 화면을 막아 두지는 않는다. 실패·지연에 대비한 최소 규칙을 둔다.

```text
호출 중      - 버튼 비활성화 + 로딩 표시(진행 중). 기존 내레이션은 그대로 유지
타임아웃     - 30초 초과 시 호출 취소, "잠시 후 다시 시도" 안내
실패         - 에러 토스트 + [다시 시도] 버튼. 기존 내용 손실 없음
재시도       - 사람이 직접 누름 (자동 무한 재시도 없음)
결과 적용     - 초안은 별도 영역에 표시, 관리자가 [적용]을 눌러야 내레이션에 반영
```

큐는 두지 않지만, 호출 실패가 편집 작업 자체를 멈추게 해서는 안 된다. AI가 죽어 있어도 관리자는 손으로 내레이션을 쓰고 승인까지 갈 수 있어야 한다.

AI는 어떤 콘텐츠도 자동 승인하지 않는다. 결과는 항상 초안이며, 최종 적용과 승인은 반드시 사람이 한다.

### MVP 이후로 미루는 AI

```text
⏸ 스토리북 검수(맞춤법 등)  - 도입하지 않음
⏸ 민감정보/개인정보 탐지     - 필요 시 추후 (외부 공유 사고 예방용)
⏸ BGM 분위기/키워드 추천
⏸ PDF 문구 생성 보조
⏸ 이미지 생성 (10장 참고)
```

## 11. 영상 패키지 내보내기 / 완성본 업로드

MemoryFlow는 영상을 직접 렌더링하지 않는다. 승인된 스토리북을 영상 제작용 패키지로 내보내고, 외부에서 제작된 완성 MP4를 다시 받아 보관·공유한다. 실제 영상 제작(렌더링) 방법은 별도 VideoFlow 문서에서 정의한다.

MemoryFlow가 책임지는 접점은 둘이다.

```text
1. 내보내기  - 승인된 스토리북을 영상 제작용 패키지로 내보냄
2. 업로드    - 외부에서 제작된 완성 MP4를 관리자가 다시 업로드
```

내보내기 패키지 구성 (MemoryFlow가 생성):

```text
project.json
scene-timeline.json   <- 장면별 길이, 사진 순서, 내레이션 텍스트
이미지/영상 파일 묶음
커버 이미지
(BGM은 Phase 2)
```

`scene-timeline.json`은 8장의 영상 길이 모델과 1:1로 대응한다. 장면 하나 = 타임라인의 한 구간. 이 패키지를 어떻게 가져와 렌더링하는지는 VideoFlow 문서의 책임이다.

> MVP에서 영상 제작 단계는 선택이다. MemoryFlow는 패키지 내보내기와 완성 MP4 업로드까지만 구현하면 된다.

## 12. 공유 링크

공유 링크는 서버에서 검증한다.

```text
/share/:token 요청
-> 토큰 해시 처리
-> share_links.token_hash 조회
-> is_active 확인
-> expires_at 확인
-> 승인된 공개 스토리북/영상 데이터만 반환
```

원본 토큰은 저장하지 않고 `token_hash`만 저장한다.

만료 옵션: 30 / 60 / 120 / 180 / 360일. 기본값 30일.

## 13. 파일 스토리지

파일은 D HDD에 마운트된 `/mnt/data`에 저장한다.

```text
/mnt/data/storage/
  projects/
    {projectId}/
      uploads/
        photos/
        videos/
      outputs/
      final-videos/
      thumbnails/
```

데이터베이스에는 파일 경로와 메타데이터만 저장한다. 파일은 공개 정적 디렉터리에서 직접 제공하지 않고, 서버 API가 세션 또는 공유 토큰 권한을 확인한 뒤 스트리밍한다. 최종 영상은 인라인 재생을 위해 HTTP Range 요청을 지원한다.

> bgm/, outputs/pdf/ 등은 해당 기능(BGM·PDF)이 들어오는 Phase 2에 추가한다.

## 15. 디자인 시스템

MemoryFlow의 디자인 토큰, 컴포넌트 규칙, 화면 패턴은 별도 디자인 시스템 문서(`docs/design-system/MASTER.md`)에서 정의한다.

PRD는 구체적인 색·폰트·치수 등 디자인 토큰을 포함하지 않는다. 디자인 관련 사항은 해당 문서를 단일 출처(single source of truth)로 따르며, 디자인이 변경되어도 PRD는 수정하지 않는다.

## 16. 배포와 운영

MVP는 로컬에서 먼저 개발하고 서버 준비는 병행한다.

서버 환경과 설치 대상(OS, 컨테이너, DB, 웹서버, AI/미디어 도구 등)의 구체적인 기술 스택은 별도 재검토 후 확정한다.

운영 원칙:

```text
MVP 배포는 수동으로 시작한다.
자동 배포(CI)는 운영이 안정된 뒤(MVP 이후) 도입한다.
백업도 MVP 이후에 정식 도입한다.
  MVP 단계에서는 최소한의 수동/일일 백업만 운영한다.
  (다단계 주기·완료 스냅샷·외부 백업은 Phase 2)
```

## 18. MVP 범위

포함:

```text
회원가입/로그인
프로젝트별 초대 링크 -> 클릭 시 자동 합류
프로젝트 생성 + Day 자동 생성
세부일정(장면) 설계
업로더별 기여 업로드 (사진/영상 + 스토리), 본인만 수정/삭제
사진 중복 보여주기 (차단 안 함)
스토리북 편집: 사진 선별, 순서, 스토리 합치기
AI 스토리 합치기 + 길이 요약 (동기 호출, 큐 없음)
영상 길이 자동 계산 (사진수 × 노출시간 + 클립 길이)
스토리북 승인 + 잠금
공유 링크 (외부 열람)
최종 영상 수동 업로드 + 접근제어 스트리밍
영상 제작용 패키지 내보내기
수동 배포
```

Phase 2:

```text
이미지 생성 (2종)
  - 프로젝트 생성 시: 입력 정보 기반 대표 이미지
  - 최종 스토리북 시: 스토리북 내용 기반 커버 이미지
민감정보/개인정보 탐지 AI
BGM 업로드/관리 + AI 추천
PDF 보고서 + AI 문구 보조
DOCX 보고서
정적 HTML 내보내기
이메일 비밀번호 재설정
이메일/카카오/SMS 알림
다단계 백업 + 완료 스냅샷 + 외부 3차 백업
자동 배포(CI) 파이프라인
```

## 19. 구현 권장 순서

```text
1. 핵심 루프
   프로젝트 생성 -> Day/일정 -> 초대 링크 -> 기여 업로드
   -> 스토리북 편집(선별/순서/스토리) -> 승인 -> 공유 링크 열람

2. AI 스토리 합치기 + 길이 요약 (동기 호출)
   + 영상 길이 자동 계산 표시

3. 최종 영상 업로드 + 영상 제작용 패키지 내보내기

4. Phase 2 기능들 (이미지 생성, BGM, PDF, 백업/CI 강화 등)
```
