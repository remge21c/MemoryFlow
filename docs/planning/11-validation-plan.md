## 0. 왜 검증부터인가

PRD v3.0은 잘 다듬어졌지만, 전체 설계가 **3개의 미검증 가정** 위에 서 있다.
이 가정이 깨지면 18장 MVP 범위 전체를 다시 그려야 한다.
**개발 비용(수 주) << 검증 비용(수 일).** 따라서 코딩 전에 검증한다.

```text
게이트 구조 (순차)
A1 (글 작성률) ── 실패 → 즉시 멈춤. A2·A4 볼 필요 없음.
   │ 통과
A2 (합치기 가치) ── 실패 → AI 합치기 핵심에서 제거, 가치 재정의
   │ 통과
A4 (영상 없는 산출물) ── 실패 → MVP 산출물 재포지셔닝 or 영상 범위 편입
   │ 통과
PRD v3.0 기반 개발 착수 (Justified)
```

## 실험 1 — A1: "업로더가 텍스트 스토리를 직접 쓴다"

> **[2026-06-03 갱신] 설계로 해소(RESOLVED BY DESIGN).**
> 업로더는 단톡방의 불특정 다수가 아니라 **지정된 책임자**이며, 글 작성은
> 자발적 참여가 아니라 **의무**다. 따라서 "글이 안 모인다(빈 입력)" 위험은
> 구조적으로 제거된다. 아래 자발적 작성률 실험은 **불필요**.
>
> **단, 잔여 가정이 A1'로 이동한다 → 실험 2 상단 참고.**
> "글이 쓰인다"는 보장돼도 "한 장면에 여러 명의 글이 모인다"는 별개이며,
> 이것이 A2(합치기 가치)의 전제다.

### (참고용·실행 보류) 원래의 자발적 작성률 실험

가장 위험한 단일 가정. 이게 거짓이면 6장(기록 모델)·7장(편집)·9장(AI)이 빈 입력을 받는다.

### 방법 (앱 없이, 실제 사람에게)
지난 행사(여행·수련회·캠프 등) 참여자 단톡방 1곳, 또는 10명에게 개별 요청.

**복붙용 메시지:**
> "지난 OO 때 찍은 사진 중 마음에 남는 거 1~2장이랑,
>  **그 순간이 어땠는지 한 줄 소감**도 같이 보내주실 수 있어요?
>  나중에 다 같이 보는 기록으로 정리하려고요 🙏"

### 측정
- 사진을 보낸 사람 중 **의미 있는 글(10자 이상, 이모지/단답 제외)을 동반한 비율**

### 판정
| 결과 | 기준 | 해석 |
|------|------|------|
| ✅ Pass | 사진 게시자의 **50% 이상**이 글 동반 | 자발적 글 작성 성립. PRD대로 진행 |
| ⚠️ Inconclusive | 25~50% | 작성 유도 설계(프롬프트/예시) 필요. 실험 1b로 |
| ❌ Fail | **25% 미만** | 자발적 글 작성 가정 붕괴 → 아래 피벗 |

### 실패 시 영향 추적 (Impact Trace)
- 무너지는 것: contribution(기여 묶음) 모델, AI 합치기, "여러 관점" 전제
- **피벗 후보:**
  1. 입력 방식 전환 — 타이핑 대신 **음성 메모 → STT**, 또는 단답형 태그/이모지
  2. 수집 주체 전환 — 관리자/총무 1명이 사진 보며 **대신 한 줄 채록**
  3. 가치 전환 — "여러 관점 스토리"가 아니라 "사진 아카이브 + 자동 자막"

### 비용 / 기간
메시지 1건, **2~3일** 내 결과.

## 실험 2 — A2: "여러 관점을 한 내레이션으로 합치는 게 진짜 가치"

> **[2026-06-03 갱신] 설계로 해소(RESOLVED BY DESIGN).**
> A1' 확인 결과: 장면당 **2~3명**이 각자 관점을 기록 → 합칠 다관점이 실재.
> 통합 방식: AI 1차 통합 초안 → **관리자가 최종 검증·다듬기**(PRD 9장과 일치).
> AI 출력이 불완전해도 사람이 확정하므로 통합 품질은 위험이 아니다.
>
> **잔여 메모(차단 아님):** 장면당 글이 2~3개·짧은 분량이면 관리자가
> AI 초안 검토보다 직접 합치는 게 빠를 수 있다. → AI의 핵심 효용은
> "통합"보다 **"장면 길이에 맞춘 요약"**(8장 길이 모델 연동)일 가능성.
> 구현 시 이 우선순위를 반영한다.

(해소됨 — 아래 종이 실험은 실행 보류)

### 방법 (종이 실험, 손으로)
1. 지난 행사의 **한 일정(장면)** 선택
2. 그 장면에 대한 **3명의 한 줄 소감** 확보 (A1 실험에서 나온 실제 글 사용)
3. 두 버전을 손으로 만든다:
   - **버전 A (합침):** 3개를 한 단락 내레이션으로 직접 작성 (AI 흉내)
   - **버전 B (개별):** 3개를 "OO: …, XX: …" 그대로 나열
4. 관리자 후보 + 참여자 **5명**에게 나란히 보여주고 질문

**복붙용 질문:**
> "같은 순간을 두 가지로 정리해봤어요.
>  A는 하나의 이야기로 합친 거, B는 각자 말 그대로 둔 거예요.
>  **최종 기록물로 남긴다면 어느 쪽이 더 좋으세요?** 이유도 한마디만."

### 판정
| 결과 | 기준 | 해석 |
|------|------|------|
| ✅ Pass | 다수가 A(합침) 선호 **AND** 관리자 "이거 내가 쓸 것 같다" | AI 합치기 가치 확인 |
| ⚠️ Inconclusive | 의견 갈림 (장면 종류에 따라 다름) | 합치기를 **선택 기능**으로 (강제 X) |
| ❌ Fail | 다수가 B 선호 / "합치면 개인 목소리가 뭉개진다" | AI 합치기 핵심에서 제거 |

### 실패 시 영향 추적
- 무너지는 것: 9장 유일 AI 기능, 7장 "스토리 합치기", 제품 차별점
- **피벗 후보:** AI는 합치기 대신 **개별 목소리 유지 + 가벼운 다듬기/순서 추천**.
  제품 정체성이 "AI 영상 대본 생성기" → "협업 스토리북"으로 이동.

### 비용 / 기간
종이/메모, **1~2일**.

## 실험 3 — A4: "영상 없는 MVP 산출물(웹 스토리북)이 만족스럽다"

> **[2026-06-03 갱신] 설계로 해소(RESOLVED BY DESIGN).**
> MVP 최종물은 **웹 스토리북**이며, 동시에 그 데이터를 **영상 제작용 back data
> (project.json + scene-timeline.json + 미디어 묶음, PRD 11장)**로 내보낸다.
> 즉 같은 데이터의 두 출력 — 사람이 보는 웹 / 기계가 먹는 패키지.
> MVP가 영상을 렌더링하진 않지만 "영상을 만들 수 있는 완결 데이터"를 산출하므로
> 영상 약속이 비어 있지 않다.
>
> **잔여 메모(차단 아님):** 내보내기 패키지 포맷이 실제 영상 도구가 소비 가능한
> 구조여야 한다. 단, 렌더링 소비 책임은 VideoFlow 문서(PRD 11장). MemoryFlow는
> 구조가 잘 잡힌 패키지 내보내기까지만 책임진다.

(해소됨 — 아래 목업 실험은 실행 보류)

### 방법 (목업 1장)
- 스토리북 웹페이지 목업 **1장** 제작 (Figma/슬라이드/손그림 무관).
  PRD 7장 "편집 화면 개념"의 열람 버전: Day → 장면 → 사진 흐름 + 합쳐진 내레이션
- 타깃 사용자 **5명**에게 제시

**복붙용 질문:**
> "행사 끝나고 **이런 웹페이지(링크로 공유)**가 최종 결과물이라면 만족스러우세요?
>  아니면 **꼭 영상으로 나와야** 의미가 있나요?"

### 판정
| 결과 | 기준 | 해석 |
|------|------|------|
| ✅ Pass | "영상 없으면 의미 없다" **30% 미만** | 웹 스토리북 단독으로 MVP 가치 성립 |
| ⚠️ Inconclusive | 30~50% | 영상은 강한 옵션. MVP는 웹, 영상은 빠른 Phase 2 |
| ❌ Fail | "영상 필수" **50% 이상** | MVP 산출물 재설계 (아래) |

### 실패 시 영향 추적
- 무너지는 것: 18장 MVP 범위의 "산출물" 정의, 11장 외부 의존 전략
- **피벗 후보:**
  1. 영상 렌더링을 MVP로 끌어옴(범위 大) — 또는 최소 슬라이드쇼 자동 생성
  2. 포지셔닝 정직화 — MemoryFlow MVP는 "영상 도구"가 아니라 "협업 스토리북"임을 명시하고 기대치 조정

### 비용 / 기간
목업 1장, **2~3일**.
